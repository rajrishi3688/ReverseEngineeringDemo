## PRE-COMPARISON NORMALIZATION (MANDATORY)

Before identifying any gaps, you MUST normalize and canonicalize both legacy and target rules:

1. Normalize syntax and structure:

   * Treat equivalent functions as the same (e.g., ROUND vs Math.Round)
   * Normalize percentage formats (0.09 vs 9%)
   * Ignore implementation-specific artifacts (file names, class names, UI labels)

2. Normalize terminology:

   * Map different variable names to same meaning (e.g., premium_base vs premiumAfterCoreRules)
   * Standardize business terms (tax, discount, fee, validation, workflow, etc.)

3. Extract and enrich each rule with:

   * country (explicit or inferred)
   * rule_type (tax, discount, fee, validation, workflow, compliance, etc.)
   * business_intent (clear human-readable purpose of rule)
   * calculation_pattern (percentage, fixed, conditional, tiered, etc.)

4. Create a canonical identity for each rule:
   (country + rule_type + business_intent + calculation_pattern)

5. Dont identify any requirements for Spain as a Gap ..

## SEMANTIC EQUIVALENCE LOGIC (CRITICAL)

A rule MUST NOT be marked as a gap if:

* Business intent is the same
* Country context is the same
* Calculation logic is materially equivalent

Treat rules as EQUIVALENT if differences are only:

* Syntax differences (ROUND vs Math.Round)
* Representation differences (9% vs 0.09)
* Structural/code differences
* Naming differences

Treat rules as DIFFERENT only if:

* Formula materially differs (e.g., 9% vs 3.2%)
* Conditions differ (e.g., applies to FAMILY vs INDIVIDUAL)
* Thresholds differ
* Sequencing differs
* Exception handling differs
* Override/config behavior differs

---

## COUNTRY-SPECIFIC MATCHING (MANDATORY)

* Always match rules within the SAME country first
* Do NOT compare rules across countries unless explicitly global

Rules:

* If a country-specific rule exists in both legacy and target:
  → Assume implemented
  → Only mark as gap if materially different

* If country missing entirely in target:
  → classify as missing_features

---

## GAP DETECTION DECISION LOGIC

For each legacy rule:

1. Find matching target rule using canonical identity
2. If match found:

   * Evaluate semantic equivalence:

     * If equivalent → DO NOT report as gap
     * If partially different → add to incorrect_implementations
3. If no match found:

   * Add to missing_features

---

## FALSE POSITIVE PREVENTION (HIGH PRIORITY)

DO NOT report a gap if:

* Equivalent rule exists in target
* Only naming, formatting, or structural differences exist
* Same logic implemented differently

When uncertain:
→ Prefer NOT marking as gap
→ Instead classify as "equivalent with variation"
